## 测试优先级与取舍考量

### 测试执行优先级排序（P0 → P2）
| 优先级 | 范围 | 理由 |
| ---- | ---- | ---- |
| P0 | 库存一致性（预占/扣减/恢复）、订单支付核心流程、幂等与重试 | 资金与库存直接相关，错误会导致超卖/资损；必须保障原子性 |
| P1 | 状态机流转（取消、退货）、WMS 异常处理（超时/乱序）、常规性能 | 影响用户体验，但错误后可通过补偿恢复 |
| P2 | 边界场景（部分发货、秒杀极端并发）、可观测性细节指标 | 发生概率低，可在迭代中逐步完善 |

---

### 取舍与折中
- 不做全场景分布式事务：预占+下单使用本地消息表 + 最终一致性，放弃 2PC 以换取高可用。
- 秒杀场景：采用 Redis 预扣 + 异步落库，允许极短时间内短暂超卖（< 0.01%），通过每日对账修复。
- 性能目标：优先保证下单接口的 P99 延迟 < 300ms，允许库存查询使用读写分离（可能会看到 1-2 秒延迟）。
- 环境依赖：WMS/SC2P 测试环境不稳定时，优先使用 Mock 验证业务逻辑，上线前做一次真实联调。
- 可观测性：先保障关键指标（库存差异率、消息积压），次要指标（JVM 内存）放到监控平台默认采集。

---

### 风险驱动测试密度
- 高风险模块（库存变更、支付回调）→ 自动化测试覆盖率要求 100%，且每个提交必须触发该模块的回归。
- 低风险模块（前端展示、搜索排序）→ 手动抽测 + 用户验收测试。

---

### 发布节奏建议
- 采用灰度发布：1% → 10% → 50% → 100%，每个阶段观察 2 小时库存差异率与错误率。
- 若差异率 > 0.1% 或出现支付回调查询超时，立即回滚并触发补偿任务。

---

### 测试数据管理
- 使用脱敏的生产数据副本（已去除个人身份信息）进行性能测试。
- 隔离 “测试账号” 的订单不进入真实 WMS，避免干扰生产库存。
